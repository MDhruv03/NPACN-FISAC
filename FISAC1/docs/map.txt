Question
Group-12: Design and implement a secure real-time Location Sharing Web Application where authenticated users broadcast live coordinates to a central server, and the server instantly distributes updates to authorized connected clients using WebSockets over TCP.

Required outcomes
1. Analyze complete TCP + WebSocket workflow: secure setup, handshake, full-duplex exchange, graceful termination, and concurrency model justification.
2. Implement concurrent authenticated server with continuous location ingestion, persistent timestamped storage, syslog events, and efficient broadcast while handling partial transmissions and latency.
3. Configure and evaluate SO_REUSEADDR, SO_KEEPALIVE, SO_RCVBUF, TCP_NODELAY; analyze latency, throughput, session behavior, and write performance.
4. Evaluate robustness under disconnects, high-frequency updates, congestion, and unauthorized attempts; analyze TIME_WAIT/CLOSE_WAIT behavior, log correctness, and DB consistency.

Implemented runtime architecture
[Frontend Browser]
        <-> WebSocket (TCP 8080)
[C Server: select()-based concurrent loop]
        <-> HTTP loopback (127.0.0.1:5000)
[Python Service Layer]
        <-> SQLite persistent storage

OUTCOME 1: TCP + WebSocket workflow and concurrency model

What we implemented
- TCP connection acceptance and WebSocket upgrade in C server event loop.
- RFC6455 handshake (Sec-WebSocket-Key -> SHA1 -> Base64 -> Sec-WebSocket-Accept).
- Full-duplex frame exchange: auth/register/location/subscribe/error.
- Graceful close handling (close frame + socket cleanup).
- Single-threaded select()-based multiplexing for concurrent clients.

Code anchors to cite in viva
- src/server.c: server_run()
    - accept new sockets, handshake, track clients[], route readable sockets.
- src/websocket.c: websocket_handshake()
    - computes Sec-WebSocket-Accept and returns HTTP/1.1 101 response.
- src/websocket.c: websocket_frame_recv()/websocket_frame_send()
    - full frame decode/encode and ping/pong handling.

Compact code evidence
```c

int activity = select(0, &readfds, NULL, NULL, &timeout);
if (FD_ISSET(server->sock, &readfds)) {
        SOCKET new_socket = accept(server->sock, ...);
        if (websocket_handshake(new_socket) == 0) {
                server->clients[i].sock = new_socket;
        }
}
```

```c
/* src/websocket.c */
snprintf(concatenated_key, sizeof(concatenated_key), "%s%s", key_start, WEBSOCKET_GUID);
sha1_update(&sha1_ctx, (uint8_t *)concatenated_key, (int)strlen(concatenated_key));
base64_encode(sha1_digest, SHA1_DIGEST_SIZE, base64_encoded);
```

Why this satisfies outcome 1
- Shows complete connection lifecycle: TCP connect -> WS upgrade -> message exchange -> disconnect cleanup.
- Concurrency justification: select() is predictable and low-overhead for I/O-bound workloads.

OUTCOME 2: Concurrent authenticated server + persistence + syslog + broadcast

What we implemented
- Authentication/registration over WebSocket messages.
- Auth gate blocks all non-auth/register traffic before login.
- Continuous location ingestion path into SQLite via backend /location endpoint.
- Event syslogs stored through backend /log endpoint.
- Broadcast fanout from one authenticated sender to other authenticated peers.

Code anchors to cite in viva

FRONTEND DEMO SCENARIOS: WHAT HAPPENS END-TO-END
==================================================

1) Start Demo Route
- Trigger: frontend button -> toggleDemoRoute() in frontend/app.js
- Client behavior:
    - disables geolocation watch
    - emits synthetic orbit coordinates every 1200 ms
    - each point sent as location frame
- Server/backend behavior:
    - normal authenticated location handling, persistence, broadcast
- Visible output:
    - marker moves in smooth loop
    - trail grows
    - TX/RX and updates/min metrics increase
- Best explanation line:
    - "This gives deterministic movement so we can demonstrate the full ingest+broadcast path even if GPS is unstable."

2) Run Burst x25
- Trigger: runBurstTest() in frontend/app.js
- Client behavior:
    - sends 25 location frames immediately around current anchor point
- Server/backend behavior:
    - frame parsing + auth checks + location inserts + broadcast fanout spike
- Visible output:
    - TX jumps by 25
    - log stream grows quickly
    - peer markers/trails update in burst
- Best explanation line:
    - "This is a controlled mini stress case to illustrate high-frequency update handling."

3) Send Malformed Payload
- Trigger: sendMalformedPayload() in frontend/app.js
- Client behavior:
    - writes deliberately broken JSON string onto WebSocket
- Server/backend behavior:
    - cJSON parse fails in handle_message(); message dropped; server stays alive
- Visible output:
    - frontend malformed counter increments
    - connection remains usable
- Best explanation line:
    - "Malformed traffic is rejected without killing the session, showing parser resilience."

4) Run Reconnect Drill
- Trigger: runReconnectDrill() in frontend/app.js
- Client behavior:
    - force-closes current socket
    - reopens socket and replays last credentials
- Server/backend behavior:
    - old connection cleaned up, new handshake/auth path executed
- Visible output:
    - reconnect counter increments
    - workflow timeline replays connect -> auth -> subscribed -> streaming
- Best explanation line:
    - "This demonstrates recovery behavior after transient disconnects."

5) Clear Trails
- Trigger: clearTrails() in frontend/app.js
- Client behavior:
    - resets local polyline arrays and distance metric
- Server/backend behavior:
    - none (visual reset only)
- Visible output:
    - map trails cleared immediately
- Best explanation line:
    - "This is a demo reset utility and does not alter server-side data."

6) Export Snapshot
- Trigger: exportSessionSnapshot() in frontend/app.js
- Client behavior:
    - builds structured JSON with metrics, backend evidence, peers, protocol traces
    - downloads geosync-session-<timestamp>.json
- Server/backend behavior:
    - none at export moment; uses latest cached UI stats
- Visible output:
    - downloadable audit artifact generated in browser
- Best explanation line:
    - "This gives us reproducible proof of live session state at demo time."

7) Geolocation denied/failure fallback (implicit scenario)
- Trigger: geolocation error callback in startSharing()
- Client behavior:
    - increments fallback counter
    - emits simulated nearby coordinates
- Server/backend behavior:
    - same location persistence+broadcast path continues
- Visible output:
    - demo still works even without browser GPS permission
- Best explanation line:
    - "The demo remains operational despite environment limitations."
